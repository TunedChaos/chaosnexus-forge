# ChaosNexus Style Guide

Thank you for contributing! To maintain consistency across the codebase and documentation, we adhere to the following strict stylistic rules.

## 1. Typography and Punctuation

- **No Em-Dashes or En-Dashes:** We must never use em-dash or en-dashes.
- **Hyphens:** Use hyphens (-) only when hyphenating words (e.g., `user-facing`, `co-worker`).
- **Standard Punctuation:** Otherwise, we use typical punctuation such as periods, commas, semicolons, colons, and parenthesis where relevant.

## 2. Git Commits and Branches

- **Semantic Commit Messages:** All git commit messages must follow standard semantic commit message rules (e.g., `feat:`, `fix:`, `docs:`, `chore:`).
- **Semantic Branch Names:** All git branches must follow semantic branch naming conventions (e.g., `feature/add-new-api`, `bugfix/fix-memory-leak`).

## 3. Code Standards

- **DRY Principle:** Strictly enforce DRY (Don't Repeat Yourself) principles. Proactively identify and refactor repetitive logic.
- **File Length:** Try to keep all code files under 750 lines by refactoring into multiple files and organizing into descriptive short named subdirectories.
- **Path Comments:** Always include a comment at the start of all code files that is the complete path from the root of the working directory to the file currently being worked in.

For more comprehensive guidelines and language-specific instructions, please refer to the primary documentation at the central `chaosnexus-website` site.
